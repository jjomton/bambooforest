# 🌲 휴먼센터 대나무숲 Product Requirements Document (PRD) v2.0
> **익명 건의·고충 관리 시스템 고도화 및 3인 협업 사양**

본 문서는 기존 MVP(v1.0) 버전을 기반으로, 3인의 개발 팀이 역할을 분담하여 기능 및 안정성을 고도화하기 위해 작성된 제품 요구사항 정의서(PRD) v2.0입니다.

---

## 1. 프로젝트 개요 (Overview)

### 1.1 배경 및 목적
기존 "휴먼센터 대나무숲 MVP(v1.0)"를 통해 수강생들이 익명으로 건의사항을 개진할 수 있는 기본 창구가 마련되었습니다. 그러나 실제 운영 결과, 대댓글(답글)을 통한 심층 토론의 한계, 부적절한 게시글 필터링의 부재, 그리고 불명확한 익명 보장 범위에 대한 우려가 확인되었습니다.

본 고도화 프로젝트(v2.0)는 **익명성의 신뢰도를 극대화**하고, **댓글/대댓글 시스템 및 건전성 필터링**을 추가하여 안전하고 활발한 소통 생태계를 구축하는 것을 목적으로 합니다. 아울러 **GitHub Actions와 Slack/Notion 연동**을 통해 3인 팀이 실무 수준의 협업 파이프라인을 훈련합니다.

---

## 2. 제품 비전 & 핵심 대상 (Target Persona)
* **일반 수강생 (Trainee):** 신분 노출에 대한 두려움 없이 학업민원 및 고충을 솔직하게 털어놓고, 동료 수강생들과 대댓글로 깊이 있게 위로와 해결책을 나누고 싶은 사용자.
* **관리자 (Admin):** 건의 사항에 대한 수용 여부 처리를 고도화하고, 부적절한 비속어가 섞인 글은 시스템이 자동으로 걸러주기를 바라며, 새 글 등록 시 실시간 메신저(Slack) 알림을 받아 빠른 처리를 하고자 하는 운영 담당자.

## 3. 고도화 기능 요구사항 (Functional Requirements v2.0)

기존 기능(F-100~F-600)에 이어 추가/고도화할 신규 사양 리스트입니다.

### 3.1 익명성 및 보안 고도화 (Anonymity & Security)
| ID | 요구사항 상세 | 우선순위 | 비고 |
| :--- | :--- | :---: | :--- |
| **F-701** | **익명 닉네임 자동 생성기:** 글/댓글 작성 시 고정된 "익명" 텍스트 대신, 동물 이름 등을 활용한 고유 닉네임 자동 조합 부여 (예: "행복한 다람쥐") | **필수 (P0)** | 24시간 동안 세션 내 동일 닉네임 유지 |
| **F-702** | **IP/식별자 단방향 솔트 해싱:** DB 테이블에 작성자를 식별할 수 있는 어떠한 단서도 남기지 않기 위해, 작성자의 식별 키를 솔트(Salt) 처리 후 SHA-256 단방향 해싱하여 임시 식별자로만 사용 | **필수 (P0)** | DB 관리자도 작성자를 역추적할 수 없도록 보장 |

### 3.2 계층형 댓글 및 대댓글 시스템 (Threaded Comments)
| ID | 요구사항 상세 | 우선순위 | 비고 |
| :--- | :--- | :---: | :--- |
| **F-801** | **대댓글(답글) 작성:** 댓글에 대댓글을 달 수 있는 2단계 계층형 구조(Thread) 지원 | **필수 (P0)** | 깊이는 2단계(댓글 - 대댓글)로 제한 |
| **F-802** | **실시간 댓글 수 갱신:** 게시글 상세조회 및 목록 화면에서 댓글/대댓글 총개수가 실시간 반영 | **필수 (P0)** | Supabase Realtime 또는 State 동기화 |

### 3.3 커뮤니티 건전성 확보 (Content Moderation)
| ID | 요구사항 상세 | 우선순위 | 비고 |
| :--- | :--- | :---: | :--- |
| **F-901** | **금칙어/비속어 자동 필터링:** 게시글 및 댓글 작성 시 백엔드에서 비속어 라이브러리를 대조하여 자동으로 마스킹(`**`) 처리 혹은 등록 제한 | **필수 (P0)** | 등록 단계에서 필터 알고리즘 가동 |
| **F-902** | **신고 시스템:** 일반 사용자가 부적절한 게시글을 신고할 수 있으며, 누적 신고가 5회 이상일 경우 목록에서 자동 블라인드(숨김) 처리 | **권장 (P1)** | DB `status` 컬럼에 `reported_count` 추가 |

## 4. 변경된 개발 범위 (In-Scope & Out-of-Scope)

### 4.1 개발 범위에 포함 (In-Scope)
* 기존 MVP 전체 기능 (게시판 CRUD, 투표, 로그인, 어드민 대시보드)
* **[NEW]** 익명 닉네임 생성기 및 단방향 솔트 해싱 로직
* **[NEW]** 2단계 댓글/대댓글 계층형 API 및 UI
* **[NEW]** 백엔드 비속어 필터 및 신고 누적 블라인드 시스템
* **[NEW]** 슬랙 연동 알림 봇 파이프라인
* **[NEW]** GitHub Actions를 활용한 자동 빌드 검증(CI) 및 Netlify 배포(CD)

### 4.2 개발 범위에서 제외 (Out-of-Scope)
* 소셜 로그인 (카카오, 네이버 등 - Supabase 이메일 가입 체계 유지)
* 첨부파일 및 이미지 다중 업로드 기능 (v2.0 범위 밖)
* 다국어 번역 지원

## 5. 3인 협업 스프린트 기반 역할 분담 (R&R) 및 브랜치 설계

모든 팀원이 설계, API 연동, UI 구현을 종단 간(End-to-End) 협업하도록 3인 체제에 맞춰 개발 역할을 매핑했습니다.

### 1) 대표님 (PM / Security / Infra)
* **담당 스펙:** 전체 협업 마일스톤 및 브랜치 통합 리드, 익명 보안 강화(F-701, F-702: 자동 닉네임 생성 및 단방향 솔트 해싱), 관리자 슬랙 알림 연동(F-1001), GitHub Actions 빌드(CI) 환경 구축.
* **배정 브랜치명:** `feature/security-infra`

### 2) 팀원 A (Core Feature - 댓글/대댓글)
* **담당 스펙:** 게시판 코어 소통 기능(F-801, F-802: 계층형 댓글 및 대댓글 시스템 구현), Supabase DB 테이블 관계 설계, CRUD API 구축 및 프론트엔드 UI 컴포넌트 개발.
* **배정 브랜치명:** `feature/comment-thread`

### 3) 팀원 B (Moderation & UX - 건전성 및 반응형)
* **담당 스펙:** 커뮤니티 정화 시스템(F-901, F-902: 비속어 필터 마스킹 및 누적 신고 자동 블라인드 처리 API/UI), 프론트엔드 전체 반응형 CSS 최적화 및 에러 화면 리팩토링.
* **배정 브랜치명:** `feature/moderation-ux`
